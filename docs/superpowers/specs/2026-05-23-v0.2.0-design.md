# claude-code-sandbox v0.2.0 — Design

**Status:** Approved
**Date:** 2026-05-23
**Owner:** Huijo Kim
**Predecessor:** v0.1.1 (`c8a087d`, `edf786a`)

## Problem

After v0.1.1's post-review hardening pass, four follow-up items remain that we can ship as a coherent v0.2.0:

1. The Postgres dump still enters via Docker build context and is baked into a per-session image layer. The image is removed on `finish`/`stop`, but the dump exists in image layers until cleanup runs. If cleanup fails or the operator inspects the image, dump contents are exposed.
2. `meta.follower_pid` is recorded for the `docker compose logs -f` follower but never killed. Failed or abandoned sessions leave a stale process and an open log fd.
3. `ComposeConfig.build_dir_db` is a dead field — passed through, never rendered in the template, kept as a vestige.
4. `sandbox status` prints raw epoch timestamps and raw Compose JSON; there's no `sandbox list`; common failures (Docker not running, S3 NoCredentialsError, missing bare repo) bubble up as Python tracebacks rather than human-readable errors.

## Goals

1. Move the Postgres dump out of Docker image layers — mirror v0.1.1's credential pattern by importing the dump into a running plain `postgres:16` container.
2. Stop the log follower explicitly during `finish`, `stop`, and failed-start cleanup. Don't leak processes.
3. Delete `build_dir_db` from `ComposeConfig` and the template/CLI sites that reference it.
4. CLI operator UX pass: human-readable timestamps, a `sandbox list` verb, friendly error wrappers for the three most common boundary failures.

## Non-goals

- **Agent egress policy** (squid allowlist, container firewall, or documented weaker guarantee). This is its own design exercise and is the headline item for v0.3.0. v0.2.0 documents the current behavior honestly but does not change it.
- **Real-Claude smoke test**. Requires CI infrastructure and an opt-in Anthropic-credentials flow. Defer to v0.3.0.
- **Integration fixture decoupling**. Cosmetic, no observed failure mode. Defer indefinitely.

## Architectural changes

### 1. DB image is generic; dump enters at runtime

Today: a per-session `sandbox-db:<sid>` image is built with the dump baked into `/docker-entrypoint-initdb.d/00_dump.dump`, restored on first init.

After: the `db` service uses a generic `postgres:16` image (no per-session build). The CLI:

1. Brings up the db container (empty).
2. Waits for `pg_isready`.
3. `docker cp <local-dump> <db-container>:/tmp/dump.dump`.
4. `docker exec <db-container> pg_restore --username postgres --dbname appdb --no-owner --no-acl --clean --if-exists --exit-on-error /tmp/dump.dump`.
5. `docker exec <db-container> shred -u /tmp/dump.dump` so even the container's overlay doesn't retain it.
6. Signals the agent that the db is ready (the agent's existing healthcheck dependency takes over).

Implications:
- `images/db/Dockerfile` and `images/db/init.sh` become unused — delete them.
- `_db_image_tag` returns `postgres:16` (no per-session tag). `_cleanup_session_images` no longer removes a db image.
- `cmd_start` orchestration adds the "db ready → restore → shred → mark ready" sequence between `compose up` and the agent-side `docker cp`.
- The `agent` service's `depends_on: db: condition: service_healthy` needs to remain — `pg_isready` is still the gate. The agent already does its real work after this gate.
- A new `docker.exec_in_container` helper joins the docker module.

### 2. Log-follower lifecycle

Add `docker.terminate_pid(pid: int) -> None` that sends SIGTERM, waits briefly, then SIGKILL. `cmd_finish`, `cmd_stop`, and the `_start_session` failure path call it on `meta.follower_pid` (and tolerate `meta.follower_pid is None`).

### 3. Drop `build_dir_db`

Delete the field from `ComposeConfig`, remove the corresponding template variable (there isn't one to render, but verify), and stop passing it in `cmd_start`. Snapshot test in `test_compose.py` retains the rest of the invariants.

### 4. CLI operator UX

- `sandbox status <id>` formats `started_at`/`finished_at` as local ISO timestamps and renders Compose state as `<service>: <state>` pairs instead of raw JSON.
- New `sandbox list` verb prints a fixed-width table: id, status, branch, started, age.
- Three error wrappers in `cli.main`:
  - `docker.DockerNotRunning` (new exception): "Docker daemon not reachable. Is Docker Desktop running?"
  - `botocore.exceptions.NoCredentialsError`: "AWS credentials not found. Set `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY` or configure `~/.aws/credentials`."
  - `LookupError` from `session.find/load`: surface the message as a clean error (currently bubbles as traceback if `cmd_finish`/`cmd_status` weren't entered).

## Data flow

`sandbox start ...` after v0.2.0:

1–7. Unchanged through `compose up` (creds copy, repo bundle, bare.git init, compose render, up, agent-side docker cp of repo+creds).

8. **New:** wait for `db` healthcheck (compose `depends_on` already gates the agent, but the CLI also blocks here so it can sequence the dump import).
9. **New:** `docker cp <local-dump> <db-container>:/tmp/dump.dump`.
10. **New:** `docker exec <db-container> pg_restore ...`.
11. **New:** `docker exec <db-container> shred -u /tmp/dump.dump`.
12. **New:** unblock the agent (the agent's `depends_on` was already satisfied by `pg_isready`, but the agent's entrypoint waits on `/input/repo.bundle` which we already cp'd before step 8 — so order matters: agent's docker cp of inputs MUST happen before agent starts work, and the dump import happens in parallel or before. Sequencing is documented in the plan.)
13. Start log follower (unchanged).

Failure handling: any exception in steps 8–11 marks the session `failed`, terminates the follower, removes images, and cleans the input dir — same pattern as v0.1.1.

## Error handling

| Failure | Behavior |
|---------|----------|
| Dump import fails | Session marked `failed`; db logs preserved; agent never starts useful work (its inputs may or may not have been cp'd — both are cleaned up). |
| Follower PID is stale | `terminate_pid` tolerates `OSError(ESRCH)` — the process is already gone. |
| Docker daemon down | `DockerNotRunning` raised by the first `docker info`-checking call in `cli.main`; user sees friendly message. |
| No AWS creds | `NoCredentialsError` caught in `cli.main`; user sees credential hint. |

## Acceptance criteria for v0.2.0

- All unit tests green; both integration tests green.
- No `sandbox-db:*` image is created during a session.
- `meta.follower_pid` process is gone after `finish` or `stop`.
- `sandbox list` and updated `sandbox status` output is human-readable.
- The three error wrappers produce the expected messages for their failure modes (tested via mocked exception injection).

## Open implementation details

- The agent's `depends_on: db: condition: service_healthy` was useful when restore happened in the db entrypoint. With runtime import, the agent could see "db healthy" before the dump is loaded. Two options: (a) hold the agent off via a different signal — e.g., the agent entrypoint already waits for `/input/repo.bundle`, so if the CLI cp's that file only AFTER dump restore, the existing wait-for-inputs in the agent serves as the gate. (b) Add an explicit ready-file the db drops after restore. Option (a) is simpler — chosen.
- `shred` may not be in the upstream `postgres:16` image. Fallback: `rm -f /tmp/dump.dump`. Verify during plan.
