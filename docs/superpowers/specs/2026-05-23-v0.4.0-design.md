# claude-code-sandbox v0.4.0 — Design

**Status:** Approved
**Date:** 2026-05-23
**Owner:** Huijo Kim
**Predecessor:** v0.3.0 (`d8fa918`)

## Problem

v0.1.0 made an early design call: copy `~/.claude/.credentials.json` from the host into the agent container, treating that file as the source of truth for the user's Claude auth. That works on Linux when the user has run `claude login` and the OAuth tokens land in the file.

It fails on macOS: the host `claude` CLI stores live OAuth state in the system Keychain, with only a stale snapshot mirrored to `~/.claude/.credentials.json`. The sandbox copies the stale snapshot; the agent's `claude` binary tries to refresh, gets a 401, and the session crashes before any work runs.

The current escape hatch is `ANTHROPIC_API_KEY` — but that's pay-per-token API billing, not a Pro/Max/Team subscription. Users with subscriptions shouldn't have to add API credits to use this tool against their existing subscription quota.

Anthropic ships the right primitive: `claude setup-token`, which mints a long-lived OAuth token from any Claude subscription. Claude Code reads that token from `CLAUDE_CODE_OAUTH_TOKEN` env. Subscription billing, no API key, no `.credentials.json` shuffling, no Keychain dance.

## Goals

1. **Support subscription auth via `CLAUDE_CODE_OAUTH_TOKEN`.** The CLI accepts it from the host env and forwards it to the agent container.
2. **Keep `ANTHROPIC_API_KEY` working as a fallback** for users on API billing.
3. **Stop shipping `~/.claude/.credentials.json` into the container.** It's brittle, OS-specific, and now obsolete.
4. **Friendly error when neither env var is set**, pointing the user at `claude setup-token`.
5. **Update the smoke test fixture** so it works with either token style without manipulating files on disk.

## Non-goals

- Auto-running `claude setup-token` for the user. It's an interactive one-time step on the host; making it automated is out of scope.
- Reading the token directly from macOS Keychain. The env-var path makes this unnecessary.
- Persisting the token in the session dir or sharing across sessions. It's just an env var passed through compose; per-session ephemerality intact.

## Architectural changes

### 1. Host-side auth resolution

A new helper `_resolve_host_auth() -> tuple[str, str]` in `sandbox.cli`:

- If `CLAUDE_CODE_OAUTH_TOKEN` is set: return `("CLAUDE_CODE_OAUTH_TOKEN", <value>)`.
- Else if `ANTHROPIC_API_KEY` is set: return `("ANTHROPIC_API_KEY", <value>)`.
- Else: raise a domain error with a clear remediation message.

Called early in `cmd_start`, before `session.new_session`, so we fail fast with no orphan state.

### 2. Compose template passes the chosen auth env

`ComposeConfig` gains:
```python
auth_env_name: str   # "CLAUDE_CODE_OAUTH_TOKEN" or "ANTHROPIC_API_KEY"
auth_env_value: str
```

The agent service's `environment:` block uses Jinja to render the chosen var:

```yaml
  agent:
    ...
    environment:
      {{ auth_env_name }}: {{ auth_env_value | tojson }}
      HTTPS_PROXY: "http://proxy:3128"
      ...
```

`{{ auth_env_value | tojson }}` ensures any quotes or special chars in the token are properly escaped.

### 3. Agent entrypoint simplification

Today's entrypoint:
1. Wait for `/input/repo.bundle` and `/input/.credentials.json` to be `docker cp`'d in.
2. Copy `.credentials.json` to `$HOME/.claude/.credentials.json`.
3. `shred -u /input/.credentials.json`.
4. Clone the bundle, run `claude`.

After v0.4:
1. Wait for `/input/repo.bundle` to be `docker cp`'d in.
2. Clone the bundle, run `claude`.

The auth env var is already in the container's env (set by compose). Claude Code picks it up. No file shuffling, no shred. The `$HOME/.claude/` setup goes away entirely.

### 4. Stop copying `.credentials.json`

`_start_session` no longer:
- Copies `~/.claude/.credentials.json` into `sdir/input/`.
- `docker cp`s it into the agent container.
- References it in `_cleanup_sensitive_build_inputs`.

The remaining input that flows via `docker cp` is just `repo.bundle`.

### 5. Smoke test fixture rewrite

The current `smoke_creds` fixture checks for `ANTHROPIC_API_KEY` or `~/.claude/.credentials.json` and tries to manipulate a fake `~/.claude/` directory. After v0.4 it just checks that one of the two real env vars is set; nothing else.

## Components

| Unit | Change |
|------|--------|
| `sandbox.cli` | New `_resolve_host_auth`; `_start_session` no longer copies creds, no longer mkdirs `input/`. Updated error message. |
| `sandbox.compose.ComposeConfig` | + `auth_env_name`, `auth_env_value` |
| `compose.yml.j2` | Agent environment block renders `{{ auth_env_name }}: {{ auth_env_value \| tojson }}` |
| `images/agent/entrypoint.sh` | Drop the creds-copy + shred + `$HOME/.claude` setup. |
| `images/agent/Dockerfile` | Drop `COPY .credentials.json` if it's there (it shouldn't be, per v0.1.1 — verify). |
| `tests/smoke/conftest.py` | `smoke_creds` checks for either env var; skips otherwise. No HOME manipulation. |
| `tests/test_cli.py` | Tests for `_resolve_host_auth`; existing creds-missing tests updated to assert the new error message. |
| `tests/test_compose.py` | Test that auth env var renders correctly under both auth modes. |

## Data flow update

`sandbox start ...` after v0.4:

1. `_resolve_host_auth()` — fails fast if neither env var is set.
2. `session.new_session`.
3. `dump.fetch`.
4. Assemble agent build context. No `.credentials.json` copy; only the agent files.
5. Bundle host repo to `sdir/input/repo.bundle` (the only thing in `input/` now).
6. Bare repo init.
7. Resolve egress allowlist, render `squid.conf` (v0.3.0).
8. Build proxy image (v0.3.0).
9. Render `compose.yml` with the chosen auth env. **Token enters the rendered YAML here.**
10. `docker compose up`.
11. `docker cp` the squid allowlist; wait for proxy healthy.
12. Wait for db ready; import dump (v0.2.0).
13. `docker cp /input/repo.bundle` into the agent container. (No creds cp anymore.)
14. Start log follower.

The token never lives in a file under the session dir. It lives in two places: the rendered `compose.yml` (which the operator owns) and the running container's env (which is `docker inspect`-visible only to operator-level access). Same blast radius as `ANTHROPIC_API_KEY` was previously.

## Error handling

| Failure | Behavior |
|---------|----------|
| Neither env var set | `cmd_start` aborts with an actionable message before any state is created. Exit code 2 (same as old creds-missing). |
| Token value is empty string | Treated as not set. |
| Token value has invalid characters for YAML | `tojson` filter handles it; rendered output stays valid. |
| Agent rejects the token at runtime | Same as today — agent exits, `cmd_finish` reports `crashed`. No special handling beyond what v0.2 already does for non-zero exit codes. |

- `test_cli.py::test_resolve_host_auth_prefers_oauth_token` — both env vars set, returns the OAuth one
- `test_cli.py::test_resolve_host_auth_falls_back_to_api_key`
- `test_cli.py::test_resolve_host_auth_raises_when_neither_set`
- `test_cli.py::test_start_aborts_when_no_auth_env_set` — updated friendly-error message includes "claude setup-token"
- `test_compose.py::test_agent_renders_oauth_token_env_when_configured`
- `test_compose.py::test_agent_renders_api_key_env_when_configured`

### Integration

The integration tests already use the stub agent which doesn't care about auth env vars; they pass through unchanged. We do need to provide a dummy `CLAUDE_CODE_OAUTH_TOKEN` in the integration test fixtures so `_resolve_host_auth` doesn't abort.

### Smoke

`tests/smoke/test_real_claude.py` runs against the real `claude` binary using whatever token the host operator has set. Re-run on a host that has `CLAUDE_CODE_OAUTH_TOKEN` set via `claude setup-token` to validate the subscription path.

## Acceptance criteria for v0.4.0

- All unit tests green.
- Both integration tests green (with dummy auth env in fixture).
- Friendly error message when neither env var is set, naming `claude setup-token`.
- No `.credentials.json` references anywhere in `src/`, `images/`, or production tests.
- README documents `claude setup-token` as the primary auth path; `ANTHROPIC_API_KEY` as the fallback.
- PROGRESS.md notes the macOS Keychain limitation as resolved.

## Open implementation details

- The integration tests' `fake_creds` fixture currently creates a fake `~/.claude/` dir. After v0.4 that's irrelevant — replace with a `fake_auth_env` fixture that sets `CLAUDE_CODE_OAUTH_TOKEN` to a dummy value via `monkeypatch.setenv`.
- The `agent` Dockerfile may have an unused `mkdir -p /root/.claude` line from v0.1.0 — strip it.
- After this change, `--credentials-json-path` (if it ever existed as a CLI flag) is dead. Check and remove.
